## Objective

`disk-agent` is being rewritten from Python to Rust to make deployment on the
Ubuntu host simple and durable.

The current Python implementation works inside the Arch Linux Podman
development environment, but it is awkward to install cleanly on the Ubuntu
24.04 host. Ubuntu follows PEP 668, so direct host installs such as
`python3 -m pip install --user -e .` are not a clean long-term answer. The
project should not rely on:

- `--break-system-packages`
- running from the Archbox development container
- fragile launcher scripts that assume Python import paths
- complex Python packaging just to run a small local system utility

The long-term target is a standalone Rust binary that can be copied or
installed onto the Ubuntu host without depending on Python at runtime. This is a
good fit because `disk-agent` is a small, read-only Linux CLI that:

- runs bounded local commands such as `df`, `du`, and optionally
  `podman system df`
- stores one JSON snapshot per day
- compares existing snapshots
- renders deterministic text reports
- never deletes, prunes, or modifies user data

Rust is preferred because it turns deployment into a single-binary concern
instead of a Python environment concern.

## Architecture

The Rust rewrite adopts three layers:

```text
CLI
  main.rs
  cli.rs

Business logic
  snapshot.rs
  report.rs
  diff.rs
  explain.rs
  investigate.rs
  classify.rs

Infrastructure
  command.rs
  filesystem.rs
  podman.rs
  json.rs
  paths.rs
  output.rs
  errors.rs
  rules.rs
  models.rs
```

### CLI layer

- `src/main.rs`
  - Parses the command selected by `clap`.
  - Dispatches to the corresponding business-logic command function.
  - Prints normal output to stdout.
  - Prints errors to stderr with the `disk-agent-rs:` prefix.
  - Converts success/failure into process exit codes.

- `src/cli.rs`
  - Defines the temporary CLI binary surface using `clap`.
  - Current subcommands are `snapshot`, `report`, `diff`, `explain`, and
    `investigate`.
  - The binary name is intentionally `disk-agent-rs` until parity is proven.

### Business logic layer

- `src/snapshot.rs`
  - Currently implements snapshot saving only.
  - Saves to `YYYY-MM-DD.json` based on the timestamp prefix.
  - Writes through a temporary file and then renames it.
  - Live snapshot collection is intentionally deferred.

- `src/report.rs`
  - Finds snapshot files.
  - Loads the latest snapshot.
  - Renders the report output to match Python semantics.
  - Implements top consumers, Podman display, largest directories, and the
    filesystem assessment text.

- `src/diff.rs`
  - Loads the latest two snapshots.
  - Compares `home_usage`, `local_share_usage`, and `copilot_usage`.
  - Preserves the 50 MiB significance threshold.
  - Sorts changes by absolute byte delta descending.
  - Renders growth and shrinkage sections.

- `src/explain.rs`
  - Renders explanations from already-loaded snapshots.
  - Reuses diff and investigation helpers.
  - Preserves the one final `Recommendation:` section behavior.
  - Does not perform live collection.

- `src/investigate.rs`
  - Implements investigation rendering and assessment logic for loaded
    snapshots.
  - Implements non-overlapping growth filtering, repeated-growth detection,
    Codex package cause text, assessment, and recommendations.
  - The live `investigate` command is intentionally deferred because it needs
    live snapshot collection.

- `src/classify.rs`
  - Matches paths against rules.
  - Preserves longest-pattern-first behavior through `rules.rs`.
  - Returns the same unknown-growth fallback semantics as Python.

### Infrastructure layer

- `src/models.rs`
  - Defines `Snapshot`, `DirectoryUsage`, `FilesystemUsage`, `PodmanUsage`,
    and `UsageChange`.
  - Uses `serde` for JSON compatibility.
  - Provides defaults for optional fields to match Python behavior.

- `src/json.rs`
  - Loads snapshots from JSON.
  - Writes pretty JSON snapshots.

- `src/paths.rs`
  - Resolves `$HOME`.
  - Builds `~/.disk-agent/snapshots`.

- `src/output.rs`
  - Implements byte formatting equivalent to the Python formatter.

- `src/rules.rs`
  - Embeds top-level `rules/*.yaml` files with `include_str!`.
  - Implements the same small YAML-like parser shape as Python instead of
    depending on a general YAML parser.
  - Sorts rules by pattern length descending.

- `src/command.rs`
  - Placeholder for future process execution abstraction.
  - Expected to own subprocess spawning, timeout behavior, stdout/stderr
    capture, exit code capture, and `LC_ALL=C`.

- `src/filesystem.rs`
  - Placeholder for future `df` and `du` collection.

- `src/podman.rs`
  - Placeholder for future `podman system df` and rootless storage fallback.

- `src/errors.rs`
  - Currently aliases `anyhow::Result`.

## Compatibility requirements

### Python snapshot compatibility

Implemented:

- Rust can read Python-generated snapshot JSON using `serde`.
- Compatibility tests cover a full current snapshot fixture and a minimal
  snapshot fixture.
- The current Rust models preserve the Python field names.

Planned:

- Validate Rust against real snapshots from the Ubuntu host before replacement.
- Add fixture tests for more real-world snapshots if schema variations appear.

### Missing optional field defaults

Implemented:

- Missing `home_usage`, `local_share_usage`, `copilot_usage`,
  `largest_directories`, and `warnings` default to empty lists.
- Missing `podman` defaults to unavailable/default `PodmanUsage`.
- Missing Podman byte fields default to `None`.
- Missing `schema_version` defaults to `1`.

These match the Python `Snapshot.from_dict` behavior.

### Atomic snapshot writes

Implemented:

- Rust snapshot saving writes to a temporary `*.json.tmp` path and renames it to
  the final daily snapshot path.
- Destination naming uses `snapshot.timestamp[..10]` as Python does.

Planned:

- Revisit durability details when live collection is added. A future hardening
  pass may consider file sync behavior, but that was not part of the Python
  implementation and should not be added casually.

### Timestamp behavior

Implemented:

- Rust saving derives the day from the first 10 characters of the snapshot
  timestamp, matching Python's daily filename behavior.

Planned:

- Live timestamp generation is not implemented yet.
- Python currently uses local-aware ISO timestamps derived from current time.
  The Rust live collector must intentionally match this before enabling
  `snapshot`.

### Rule loading behavior

Implemented:

- Top-level `rules/*.yaml` files were copied from `disk_agent/rules/`.
- Rust embeds them with `include_str!`, so the future binary does not require
  external rule files at runtime.
- The parser intentionally mirrors the Python subset parser rather than using a
  full YAML parser.
- Rules are sorted by descending pattern length before matching.
- Unknown paths return the same unknown-growth fallback.

Planned:

- Add explicit tests for every rule file and every known classification if rule
  behavior expands.

### Existing JSON compatibility

Implemented:

- Rust reads existing snapshot JSON.
- Rust writes pretty JSON with deterministic struct field order.

Known difference:

- Python uses sorted JSON keys when saving snapshots. Rust does not currently
  sort keys globally. This is acceptable for machine compatibility but may make
  textual diffs look different.

### Read-only safety guarantees

Implemented:

- No cleanup, delete, prune, or mutation behavior exists in the Rust scaffold.
- The implemented commands only read snapshots and render text, except
  `save_snapshot`, which writes snapshot JSON when called by tests or future
  collection code.
- The live `snapshot` and `investigate` commands currently return explicit
  phase-1 unsupported errors rather than performing partial live behavior.

Planned:

- Future live collection must remain read-only: `df`, `du`, Podman reporting,
  and rootless storage inspection only.
- Permission/read errors should become warnings, not destructive remediation.

## Remaining implementation

Recommended order:

1. Implement `command.rs`.
2. Implement `filesystem.rs`.
3. Implement `podman.rs`.
4. Wire live orchestration in `snapshot.rs`.
5. Wire live `investigate` in `investigate.rs`.
6. Run side-by-side Python/Rust output comparison.
7. Validate the compiled binary on Ubuntu.
8. Only after parity, consider renaming/replacing `disk-agent`.

This order was chosen because live snapshot collection depends on reliable,
testable infrastructure. `snapshot.rs` should orchestrate collection, not own
all collection details. Implementing the command runner first keeps process
execution behavior centralized and makes `df`, `du`, and Podman collection
testable without shelling out from business logic.

Expected responsibilities for remaining modules:

- `command.rs`
  - Spawn commands.
  - Capture stdout and stderr.
  - Capture exit status.
  - Apply timeout behavior equivalent to Python's 120-second timeout.
  - Set `LC_ALL=C`.
  - Return structured output without panicking.

- `filesystem.rs`
  - Run and parse `df -B1 --output=source,size,used,avail,pcent,target /`.
  - Run and parse `du -x -B1 --max-depth=N <path>`.
  - Preserve partial `du` output when `du` exits nonzero for permission/read
    errors.
  - Return warnings instead of failing for optional missing or unreadable paths.
  - Preserve display path behavior such as `~` and `~/...`.

- `podman.rs`
  - Prefer `podman system df --format "{{json .}}"` when `podman` exists.
  - Parse line-delimited JSON output.
  - Preserve decimal unit parsing for Podman sizes such as `kB`, `MB`, and `GB`.
  - Fall back to rootless storage inspection under
    `~/.local/share/containers/storage` when the binary is unavailable.
  - Report unavailable Podman as data, not as a fatal error.

- `snapshot.rs`
  - Orchestrate filesystem, directory, Copilot, local share, largest directory,
    and Podman collection.
  - Preserve warning aggregation.
  - Preserve largest directory deduplication and top-100 behavior.
  - Save one daily JSON snapshot.

- `investigate.rs`
  - Load the latest saved snapshot.
  - Collect a fresh in-memory snapshot.
  - Save the fresh snapshot only if today's file does not already exist.
  - Preserve the current informational-only investigation output.

## Design decisions

### Keep Python as the oracle

Decision: leave the Python implementation untouched and use it as the behavior
oracle until Rust parity is proven.

Reasoning: the Python implementation already has known behavior, tests, and
real snapshot history. Replacing it before parity would risk subtle changes in
snapshot interpretation, report output, and safety behavior.

Rejected alternative: migrate command-by-command by mixing Python and Rust at
runtime. This was rejected because it preserves the Python deployment problem.

### Temporary binary name `disk-agent-rs`

Decision: the Rust binary is named `disk-agent-rs`.

Reasoning: this avoids accidentally shadowing the working Python `disk-agent`
command before parity is proven.

Rejected alternative: immediately build a Rust binary named `disk-agent`. This
was rejected because it makes testing and rollback less clear.

### Three-layer architecture

Decision: split the Rust code into CLI, business logic, and infrastructure.

Reasoning: live collection is Linux-specific and should be isolated from report,
diff, explain, and investigation logic. This keeps business logic testable with
fixtures and prevents direct shell execution from spreading throughout the code.

Rejected alternative: port the Python layout literally with most collection
logic in `snapshot.rs`. This was rejected because the Rust rewrite is an
opportunity to make command execution and OS-specific collection explicit
without changing behavior.

### Implement JSON compatibility first

Decision: first prove Rust can read Python snapshots before implementing live
collection.

Reasoning: reading existing snapshots is the critical compatibility boundary.
It lets report/diff/explain be tested before any live command execution exists.

Rejected alternative: port live collection first. This was rejected because it
would make debugging harder: failures could come from command execution,
parsing, JSON, or rendering all at once.

### Embed rules in the binary

Decision: copy rules to top-level `rules/` and embed them with `include_str!`.

Reasoning: the deployment goal is a single binary. Embedded rules avoid another
runtime asset path and another installation step.

Rejected alternative: load external rule files at runtime. This was rejected
for phase 1 because it complicates deployment. It can be reconsidered later if
user-editable rules become a requirement.

### Preserve the small rule parser

Decision: port the Python YAML-like subset parser rather than adding a YAML
crate.

Reasoning: the current rule format is simple and the behavior is already known.
A general YAML parser could accept or normalize content differently, causing
unnecessary migration drift.

Rejected alternative: use `serde_yaml` or another YAML parser. This was
rejected for now to keep dependencies small and behavior close to Python.

### Use a small crate set

Decision: use `clap`, `serde`, `serde_json`, `anyhow`, and `tempfile`.

Reasoning: these cover CLI parsing, JSON compatibility, application-level
errors, and temporary-directory tests without introducing runtime complexity.

Rejected alternatives: async runtimes, logging frameworks, database crates, or
larger system libraries. These do not provide value for the current bounded CLI.

### Pretty JSON but not sorted keys

Decision: Rust writes pretty JSON in struct field order, not globally sorted
keys.

Reasoning: machine compatibility depends on field names and defaults, not key
order. Stable struct order is enough for readable output in phase 1.

Rejected alternative: add extra machinery to sort JSON keys exactly like
Python. This was rejected as unnecessary unless textual snapshot diffs become a
hard requirement.

### Defer live `snapshot` and `investigate`

Decision: `snapshot` and live `investigate` return explicit phase-1 unsupported
errors.

Reasoning: partial live collection would be misleading. It is safer to complete
and test `command.rs`, `filesystem.rs`, and `podman.rs` first.

Rejected alternative: implement incomplete live collection directly in
`snapshot.rs`. This was rejected because it would violate the desired module
responsibilities and increase parity risk.

- The current tests use fixtures, not the real host snapshot history.
- The Rust code currently has no end-to-end comparison test that invokes both
  Python and Rust and diffs their output.

## Risks

Highest-risk areas for migration drift:

- `du` behavior on permission errors. Python keeps partial output and records
  warnings; Rust must do the same.
- Path display behavior. Python maps paths under home to `~` and `~/...`.
- Timestamp generation. Rust must match the date and local/offset behavior well
  enough that daily filenames stay compatible.
- Podman size parsing. Python parses Podman human-readable decimal units from
  JSON lines.
- Podman fallback. Python checks rootless storage when the `podman` binary is
  unavailable.
- Rule parsing. The files look like YAML, but Python parses a small subset.
- Investigation non-overlap filtering. This logic is easy to subtly change.
- Exact report/explain wording. The output is human-facing and should remain
  stable unless intentionally changed.
- Linux binary compatibility. A binary built on Arch may not run on Ubuntu if
  linked against newer system libraries. Build or validate release binaries in
  an Ubuntu-compatible environment.

Safety risks:

- Future contributors must not add cleanup, prune, or delete commands.
- Future live collection must remain read-only.
- Optional missing paths and unreadable locations should not cause destructive
  suggestions or automatic remediation.

## Session summary

The session began with a technical evaluation of whether `disk-agent` should be
rewritten from Python to Rust. The conclusion was yes: the project is a small
read-only Linux utility, and the main active problem is deployment on Ubuntu
rather than application complexity. Python works in the Archbox development
environment, but Ubuntu 24.04's PEP 668 behavior makes host installation
awkward without unacceptable options such as `--break-system-packages` or
fragile wrapper scripts.

The existing Python implementation was inspected and treated as the behavior
oracle. Important behavior was identified: snapshots live under
`~/.disk-agent/snapshots/YYYY-MM-DD.json`, missing optional fields default
gracefully, `du` and Podman read errors are warnings rather than fatal failures,
diff ignores changes smaller than 50 MiB, explain has exactly one final
recommendation section, and investigation is informational only.

A Rust scaffold was then created beside the Python implementation. The
temporary binary was named `disk-agent-rs` to avoid replacing the Python command
before parity. The requested layered architecture was adopted so CLI parsing,
business behavior, and Linux-specific infrastructure stay separate.

Phase 1 intentionally focused on compatibility before live collection. Rust
models, JSON load/save, report rendering, diff rendering, explain rendering,
classification, and loaded-snapshot investigation rendering were implemented.
Live command execution, filesystem collection, Podman collection, live
`snapshot`, and live `investigate` were deferred because they depend on a
careful command runner and OS-specific parsing.

Compatibility tests were added using Python-format snapshot fixtures, including
a minimal snapshot that proves optional defaults. Report, diff, snapshot save,
and investigation tests were added. The Rust test suite passed with
`CARGO_TARGET_DIR=target cargo test`, and a CLI smoke test proved
`disk-agent-rs report` can read a fixture snapshot from a temporary HOME.
